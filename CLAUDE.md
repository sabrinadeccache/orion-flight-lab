# CLAUDE.md — Orion Flight Lab

Este arquivo é a documentação normativa do projeto para qualquer agente (humano ou IA) que trabalhe neste repositório. Ele deve ser lido e respeitado integralmente antes de qualquer alteração.

## 1. Visão geral do projeto

**Orion Flight Lab** é uma plataforma proprietária de sistema de registro (system-of-record) para um **CTAC — Centro de Treinamento de Aviação Civil**, certificado pela **ANAC** (Agência Nacional de Aviação Civil) sob o regulamento **RBAC 142** e a **IS 142-001**.

O sistema cobre todo o ciclo de vida de um CTAC: organização e satélites/remotos, corpo docente (instrutores e examinadores), grade curricular, matrícula e histórico acadêmico de alunos, exames teóricos/práticos, emissão de certificados, controle de qualificações e vencimentos, repositório de documentos regulatórios com versionamento, Sistema de Gestão da Qualidade (SGQ), Sistema de Gerenciamento da Segurança Operacional (SGSO), CRM/comercial, contratos e financeiro.

Trata-se de um sistema **auditável**: toda ação relevante deve deixar rastro (audit log imutável), todo prazo regulatório deve ser monitorado ativamente, e nenhuma regra de negócio (RN) pode ser contornada por composição de chamadas de API.

## 2. Stack técnica

- **apps/web** — Next.js 14 (App Router) + TypeScript + Tailwind CSS
- **apps/api** — NestJS (monólito modular) + TypeScript, módulos em `apps/api/src/modules`
- **Banco de dados** — Supabase (Postgres + Auth + Storage + Row Level Security), acessado via **Prisma ORM**
- **Filas/Jobs** — BullMQ + Redis (Upstash/Railway em produção; `docker-compose` com Redis local apenas para desenvolvimento)
- **Observabilidade** — Sentry (`@sentry/nestjs` na API, `@sentry/nextjs` no web; código real, ver §11)
- **Monorepo** — npm workspaces (sem Turborepo/Nx nesta fase)

## 4. Multi-tenant e RLS

- Toda tabela (exceto `Organization` e tabelas de junção/auditoria puras) possui `organization_id`.
- RLS habilitado em **todas** as tabelas de negócio.
- Política padrão `organization_isolation`: `auth.jwt()->>'organization_id' = organization_id::text`.
- `audit_log` é **somente inserção**: política `audit_log_insert_only` permite `INSERT`, bloqueia `UPDATE`/`DELETE` via RLS (`USING (false)`) e `REVOKE` de privilégios de update/delete do papel autenticado.
- Todo bucket de Storage é privado e as policies escopam acesso por `organization_id` embutido no path do objeto.

## 5. Regras de negócio (RN) — não negociáveis

- **RN-05** — Certificado só pode ser emitido se os requisitos do curso estiverem completos (frequência sempre + os exames que a `modality` do `Course` exigir: `TEORICO` pede exame teórico aprovado, `PRATICO` pede exame prático aprovado, `MISTO` — default, comportamento original — pede os dois). Quando o curso é `TEORICO`, emitir o certificado **cria automaticamente** a `Qualification` do aluno (`qualification_code` = código do curso, `certificate_id` vinculado). `PRATICO` não tem esse caminho automático — a qualificação correspondente é lançada manualmente via `POST /qualifications` (`AcademicService.createQualification`).
  - **Emissão automática**: `Course.min_passing_score` (opcional, 0-100) define o aproveitamento mínimo do curso. Quando um exame é registrado com `score` e sem `result` explícito, `AcademicService.registerExam` deriva `APROVADO`/`REPROVADO` comparando a nota com esse mínimo (sem `min_passing_score` configurado, cai no comportamento antigo: só `PENDENTE` até alguém informar `result` manualmente). Assim que um exame ou uma frequência é registrado e os requisitos da RN-05 passam a estar completos, o certificado (e, para `TEORICO`, a qualificação) é emitido **automaticamente** — não é mais preciso chamar `POST /certificates` manualmente. Esse endpoint continua existindo (emissão retroativa, cursos sem `min_passing_score`), mas agora **bloqueia emissão duplicada**: `BadRequestException` se o enrollment já tiver um certificado.
- **RN-07** — Aluno em quarentena de fraude (12 meses) não pode ter novos exames registrados.
- **RN-11** — Turma não pode ultrapassar 25 alunos matriculados simultaneamente.
- **RN-13** — Job diário atualiza status de qualificações/matrículas vencidas.
- **RN-15** — Instrutor não pode ministrar mais de 8h de aula em uma janela de 24h.
- **RN-16** — Proficiência anual do instrutor segue a regra dos 45 dias (renovação até 45 dias antes/depois do vencimento).
- **RN-17** — Instrutor pode ter no máximo 2 qualificações de tipo de aeronave simultâneas.
- **RN-18** — Examinador pode ter no máximo 2 credenciamentos ANAC de tipo simultâneos.
- **RN-20** — Curso inativo é alertado aos 150 dias e suspenso automaticamente aos 180 dias.
- **RN-22** — Comunicações obrigatórias à ANAC devem ser sinalizadas com 60 dias de antecedência.
- **RN-24** — Toda ação de escrita relevante gera registro em `audit_log` (imutável).
- **RN-25** — Não-conformidade (SGQ) só pode ser fechada quando houver pelo menos uma ação corretiva vinculada e todas estiverem concluídas.
- **RN-26** — Ação corretiva (SGQ) não pode ser criada com `due_date` no passado.
- **RN-27** — Risco (SGSO) de nível alto (`probability × severity ≥ 15`, matriz 5×5) não pode ser aceito/mitigado sem ao menos uma mitigação registrada.
- **RN-28** — Ocorrência de segurança (SGSO) de severidade "alta"/"critica" exige um `Hazard` vinculado para investigação reativa do SGSO.
- **RN-29** — Subscription só pode ser criada contra um Contract com `status = "ativo"`.
- **RN-30** — Pipeline não pode ir para o estágio "ganho" se a Proposal vinculada estiver com `valid_until` vencido ou `status = "recusada"`.
- **RN-31** — Payment não pode fazer o total pago exceder o `amount` da Charge; Charge muda automaticamente para `status = "paga"` quando totalmente quitada.
- **RN-32** — Job diário marca Charges vencidas e ainda `"pendente"` como inadimplentes, criando/atualizando `Delinquency.days_overdue`.

## 8. Regras de Git

- Commits em **inglês**, no padrão **Conventional Commits** (`feat:`, `fix:`, `chore:`, `docs:`, `refactor:`, `test:`).
- Commits **granulares por módulo** — nunca um commit único gigante ao final.
- Nenhum commit pode ser criado com lint ou typecheck quebrado (**zero tolerância**).
- `main` é a branch padrão e protegida por convenção; push direto é permitido neste estágio do projeto, mas todo push deve ser precedido de lint+typecheck limpos. Desde que o CI existe (ver §17), o mesmo vale para ele — não empurrar código que quebre `.github/workflows/ci.yml`.

Corrigido passando `{} as SupabaseAdminService` como terceiro argumento, mesmo padrão já usado para `StorageService`. Lição: ao mudar a assinatura de um construtor de service, checar também `apps/api/test/*.integration.spec.ts` (não só os `*.spec.ts` unitários, que usam mocks e não sentem a mudança de assinatura da mesma forma).
17. **`supabase start` roda `supabase/seed.sql` automaticamente, antes do Prisma criar qualquer tabela** — achado na primeira rodada real do novo job `test-e2e` no GitHub Actions (2026-07-19): o job falhava em "Start local Supabase stack" com `relation "public.organizations" does not exist (SQLSTATE 42P01)`. O plano original já tinha identificado esse problema de ordem para `supabase/migrations/*.sql` (por isso o rename pra `supabase/rls/`, ver `supabase/rls/README.md`), mas não para `supabase/seed.sql` — o CLI aplica o seed automaticamente durante o próprio `supabase start`/bootstrap, no mesmo momento (schema ainda vazio), independente de qualquer step de CI vir depois. Corrigido com `[db.seed] enabled = false` em `supabase/config.toml`; quem quiser a organização de demo localmente roda `psql "$DIRECT_URL" -f supabase/seed.sql` manualmente depois de `prisma migrate deploy`, mesmo padrão já usado pra `supabase/rls/*.sql`. Lição: ao desenhar a ordem de bootstrap de uma stack Supabase local a partir de um schema gerenciado pelo Prisma, checar **todo** artefato que o CLI aplica sozinho no primeiro `start` (migrations *e* seed), não só o mais óbvio.
18. **`supabase/rls/0005_security_advisor_fixes.sql` referenciava uma função (`rls_auto_enable()`) que só existe no projeto Supabase hospedado** — achado na segunda rodada do job `test-e2e` (2026-07-19, depois do fix do item 17), falhando com `function public.rls_auto_enable() does not exist`. Essa função é instalada pela própria plataforma Supabase (um helper interno de event trigger), não por nenhuma migration nossa — nunca existiu num stack local/self-hosted (`supabase start`), só no projeto real. Corrigido envolvendo os três `revoke execute` num bloco `do $$ ... if exists (select 1 from pg_proc ...) then ... end if; end $$;`, que vira um no-op inofensivo localmente e continua aplicando os revokes de verdade no projeto hospedado. Lição: SQL de "security advisor fixes" tende a referenciar objetos específicos da plataforma gerenciada — ao rodar essas migrations num Postgres genérico (local, CI, ou qualquer self-hosted), sempre guardar contra objetos platform-specific não existirem, não assumir que todo SQL commitado em `supabase/` é portável por padrão.
19. **`PORT: 3001` declarado no nível do job `test-e2e` vazou pro processo do `apps/web`, não só pro `apps/api`** — achado na terceira rodada do job (2026-07-19): `apps/api` crashava com `EADDRINUSE: address already in use :::3001`, e o `web.log` mostrava `next start` de pé em `http://localhost:3001` (não `:3000`) — o `PORT` do nível do job (pensado só pra API) é herdado por **todos** os steps do job, incluindo o `nohup npm run start --workspace=apps/web`, e `next start` também respeita `process.env.PORT`. Corrigido removendo `PORT` do bloco `env:` do job inteiro — `apps/api/src/main.ts` já tem fallback pra `3001` sem a variável (mesmo fallback documentado no item da Railway, ver armadilhas), e `next start` já usa `3000` como default sem ela. Lição: variáveis de ambiente no nível do job do GitHub Actions são globais a todos os steps/processos daquele job — nunca declarar ali algo que só faz sentido pra **um** dos processos que o job sobe; escopar no step específico (ou, melhor ainda quando existe um default correto no próprio código, simplesmente omitir a variável).
20. **`supabase status -o env` imprime valores entre aspas duplas literais (`SERVICE_ROLE_KEY="eyJ..."`), e `>> "$GITHUB_ENV"` não faz parsing de shell** — achado na quarta rodada do job `test-e2e` (2026-07-19): `apps/web/e2e/global-setup.ts` falhava com `invalid JWT: ... illegal base64 data at input byte 0` ao chamar `admin.auth.admin.createUser`, porque o `SERVICE_ROLE_KEY` recebido pelo step seguinte incluía as aspas literais como parte do valor (o arquivo apontado por `$GITHUB_ENV` é `KEY=VALUE` cru, sem parsing de aspas de shell). Corrigido com um `sed -E 's/^([A-Z_]+)="(.*)"$/\1=\2/'` entre o `supabase status -o env` e o `>> "$GITHUB_ENV"`. Lição: nunca assumir que a saída de uma ferramenta de terceiros pensada pra `eval`/shell (aspas incluídas) é compatível de graça com o formato `KEY=VALUE` puro que o `$GITHUB_ENV` do GitHub Actions espera.
21. **`/clients/new` tinha `<label>` sem `htmlFor`/`id` associando ao `<input>`** — achado na quinta rodada do job `test-e2e` (2026-07-19): login e navegação funcionaram (as duas asserções de `auth.spec.ts` passaram), mas `golden-path.spec.ts` travava em `page.getByLabel('Nome').fill(...)` com timeout de 30s — o campo estava visível na tela (confirmado no snapshot de erro do Playwright), mas sem associação de acessibilidade `label`↔`input` (diferente de `/login`, que já usa `htmlFor`/`id`), `getByLabel()` nunca o encontra. Corrigido adicionando `htmlFor`/`id` nos três campos de `/clients/new` (`name`, `cnpjCpf`, `type`). **Gap mais amplo, não corrigido nesta rodada**: esse padrão de `<label>` sem associação provavelmente se repete na maioria dos formulários do app (só `/login` foi escrito com `htmlFor`/`id` desde o início) — não foi corrigido em todo lugar de propósito, porque o escopo aqui era destravar o smoke test em `/clients/new`, não uma varredura de acessibilidade no app inteiro; fica como um item de acessibilidade pra uma rodada futura se o escopo do E2E crescer pra outras telas.

## 14. Armadilhas conhecidas (não redescobrir)

- **JWT de teste não pode ser assinado à mão** (`jwt.sign(payload, SUPABASE_JWT_SECRET)`) — desde o bug #5 acima, o guard só aceita tokens reais emitidos pelo Supabase Auth (ES256 via JWKS). Para testar endpoints protegidos, sempre: criar usuário real via `supabase.auth.admin.createUser({ app_metadata: { organization_id, roles } })` e depois `supabase.auth.signInWithPassword(...)` (client `anon`) para pegar um `access_token` de verdade.
- **`SUPABASE_URL`** deve ser só a URL base do projeto (`https://xxx.supabase.co`), nunca com `/rest/v1/` no final.
- **`DATABASE_URL`** (pooler, porta 6543) precisa de `?pgbouncer=true`. **`DIRECT_URL`** (porta 5432) não.
- Prisma bypassa RLS (conecta como owner) — a defesa multi-tenant real está 100% na camada de aplicação (`organization_id` explícito em todo `where`), não na RLS. RLS é defesa em profundidade para acesso direto via REST/anon key, não a linha de defesa principal do backend.
- `npm run dev:api` / `dev:web` fazem `npm run build:shared` antes (necessário — `@orion/shared` precisa estar compilado em `dist/`).
- **`service_role` do `supabase-js` não tem GRANT de `INSERT` em `user_profiles`** (`permission denied for table user_profiles`, código `42501`) — a API real nunca bate nesse problema porque o Prisma conecta como owner do banco, não como `service_role`. Mas se você for criar um `user_profile` de teste manualmente (§15, passo 2), **não use `admin.from('user_profiles').insert(...)` do client `service_role`** — insira via `execute_sql` (roda como owner) em vez disso.
- **`POST /exams` tem duas casing diferentes no mesmo DTO**: `type` valida contra o enum de `@orion/shared` (`teorico`/`pratico`, minúsculo), mas `result` valida contra o enum do Prisma (`APROVADO`/`REPROVADO`/`PENDENTE`, maiúsculo) — mandar `type: 'TEORICO'` retorna 400 ("must be one of the following values: teorico, pratico"). Não é bug, é só a convenção existente; ao escrever testes/scripts de exame, usar minúsculo em `type` e maiúsculo em `result`.
- **Não existe endpoint pra criar `Segment`/`Module`/`Unit`/`SubUnit`/`Lesson`** — só `TrainingProgram`, `Curriculum` e `Course` têm API (`training.controller.ts`). Pra testar `Attendance` (que exige um `lesson_id` real) ponta a ponta, a hierarquia inteira precisa ser inserida via `execute_sql` primeiro.
- **`storage.objects` do Supabase não aceita `DELETE` direto via `execute_sql`** (`ERROR: 42501: Direct deletion from storage tables is not allowed`) — é uma proteção deliberada do Supabase contra perda acidental de dado por objeto órfão. Pra apagar um arquivo de teste de um bucket privado, use a Storage API (client `supabase-js` com service role, `.storage.from(bucket).remove([path])`) — não há um MCP de Storage nesta sessão, então arquivos de teste às vezes ficam órfãos no bucket (inofensivo: são só bytes sem nenhuma linha de `DocumentVersion`/`Certificate` apontando pra eles, mas vale saber que `execute_sql` não é o caminho pra limpar isso).
- **`reactStrictMode` estava ligado (`apps/web/next.config.js`) e causava flicker visível em `next dev`**: o StrictMode do React remonta cada componente uma vez de propósito (mount → unmount → mount) só em dev, o que reseta o estado entre as duas passagens — qualquer tela que busca dado num `useEffect` (praticamente todo `new`/`[id]/edit` do app) mostrava "Carregando..." → conteúdo → "Carregando..." de novo → conteúdo, porque a requisição disparava duas vezes (confirmado via rede: `GET /students/:id` chegava 2x ao abrir a edição). Desligado (`reactStrictMode: false`) — não afeta build de produção, que nunca faz esse duplo-disparo independente da flag.
- **"Cannot find the middleware module" / "missing required error components, refreshing..."** — sintoma de cache de build (`apps/web/.next`) corrompido/desatualizado, tipicamente depois de reiniciar o `next dev` várias vezes numa sessão longa (`middleware-manifest.json` fica com `"middleware": {}` vazio mesmo com `src/middleware.ts` existindo). Uma tentativa de mitigação automática (`apps/web/scripts/check-next-cache.js` como `predev`) foi adicionada e depois **revertida a pedido do usuário** — o script não resolveu o problema de verdade (o processo antigo do `next dev` continuava rodando com o cache já carregado em memória, então apagar a pasta não bastava) e o usuário preferiu voltar ao comportamento original, sem "mágica" escondida no `predev`. Fix manual (o único caminho agora): **matar o processo do `next dev`** (`lsof -nP -iTCP:3000 -sTCP:LISTEN` pra achar o PID, `kill <PID>`), apagar `apps/web/.next` (`rm -rf apps/web/.next`), e só então subir `dev:web` de novo. Apagar só a pasta sem matar o processo não resolve — o processo antigo mantém o cache errado em memória e regenera o mesmo estado quebrado.
- **`window.open(url)` chamado depois de um `await` não é confiável** — o botão de download de Documentos fazia `await fetch(...)` pra pegar a signed URL e só então chamava `window.open`; como isso já não conta mais como parte do clique original do usuário, o bloqueador de popup do navegador barra silenciosamente (de forma intermitente, dependendo da velocidade da resposta). O padrão correto é abrir a aba **antes** do `await` (síncrono, dentro do handler de clique) e só setar `location.href` dela depois que a URL chegar — ver `handleDownload` em `apps/web/src/app/documents/[id]/edit/page.tsx`.
- **`supabase.auth.admin.inviteUserByEmail(email, { data })` só grava em `user_metadata`, nunca em `app_metadata`** — e é `app_metadata` que o `SupabaseAuthGuard` lê pra extrair `organization_id`/`roles` do JWT (mesma armadilha do bug #5 do §13, agora reencontrada num Admin API call diferente). Pra convidar um usuário já com `organization_id`/`roles` corretos, é preciso uma segunda chamada, `admin.updateUserById(userId, { app_metadata: {...} })`, logo depois do convite — ver `SupabaseAdminService.inviteStudent`.
- **`inviteUserByEmail` rejeita domínios de e-mail não roteáveis** (`example.com`, etc.) com "Email address ... is invalid" — diferente de `admin.createUser`, que aceita qualquer formato sintaticamente válido sem checar deliverability. Pra testar o fluxo de convite (`POST /students/:id/invite`) contra a infra real, use um domínio de verdade (ex.: um endereço `@mailinator.com`, cuja caixa pública dá pra conferir sem precisar de conta) em vez de `@example.com`.
- **Testar uma sessão de `ALUNO` sem depender do e-mail de convite chegar**: depois do convite, dá pra pular o clique no link usando o Admin API pra setar senha e confirmar o e-mail direto — `admin.updateUserById(userId, { password: '...', email_confirm: true })` — e então `signInWithPassword` com o client `anon`, exatamente como o fluxo de staff do §15 passo 3. Sem `email_confirm: true` o login falha com `email_not_confirmed`, já que `inviteUserByEmail` não confirma o e-mail sozinho (só o clique no link confirmaria, numa invocação real).
- **`npm run test:integration --workspace=apps/api` roda contra o Supabase real se você não prestar atenção** — o `PrismaClient` gerado carrega `.env` sozinho (comportamento nativo do Prisma, procura o arquivo a partir do diretório do `schema.prisma`/raiz do projeto), então mesmo sem exportar `DATABASE_URL` na sessão do shell, `new PrismaService()` dentro do teste de integração conecta no pooler real do Supabase (`oelepfnnnwchmubhazad`) usando o `DATABASE_URL` do `.env` da raiz — o `describeIfDatabase = process.env.DATABASE_URL ? describe : describe.skip` do §17 **não protege contra isso**, porque `process.env.DATABASE_URL` está sim populado (só não foi *você* quem populou). Confirmado nesta sessão: rodar o comando sem nenhuma env var explícita executou os 2 testes de `multi-tenant.integration.spec.ts` de verdade contra o projeto real — o `afterAll` limpou certinho (`Multi-tenant Test Org A/B` criadas e apagadas, confirmado via `execute_sql`, nenhum resíduo), mas foi sorte de o teste ser bem-comportado, não uma proteção do sistema. **Antes de rodar `test:integration` localmente**, ou setar explicitamente `DATABASE_URL`/`DIRECT_URL` para um Postgres efêmero real, ou renomear/mover o `.env` da raiz temporariamente — nunca assumir que "não exportei a variável" significa "não vai conectar no banco real". Ver §16 item 6 (gap aceito, gap ainda não fechado com um guard de código).
- **Nem todo endpoint que "parece" mapear pra uma entrada de `ROUTE_ROLES` (`apps/web/src/middleware.ts`) é consumido pela rota daquele prefixo** — ao fazer o retrofit de RBAC no backend (§16 item 5), o primeiro instinto foi restringir `GET /reports/dashboard-summary` a `ADMIN`/`GERENTE_QUALIDADE`/`GERENTE_SEGURANCA` porque existe uma entrada `'/reports'` em `ROUTE_ROLES` com exatamente esses papéis. Mas esse endpoint na verdade alimenta o `/dashboard` (home geral, sem restrição de papel) — a página `/reports` é um placeholder estático que não faz nenhuma chamada de API. Antes de aplicar `@Roles` em qualquer endpoint compartilhado, `grep` no frontend por quem de fato consome a rota (`grep -rl 'nome-do-endpoint' apps/web/src/app`), não só assumir pelo nome do módulo/prefixo.
- **Railway: o passo de install do builder ("Railpack") roda antes do Build Command e é escopado ao Root Directory, sem visibilidade do monorepo se o Root Directory for `apps/api`** — a primeira tentativa de configurar o serviço `@orion/api` na Railway usou Root Directory = `apps/api`, e o `npm install` automático que a Railway roda antes de qualquer Build Command customizado ficou restrito a esse diretório, sem enxergar o `package.json`/workspaces da raiz. Como `@orion/shared` é um pacote de workspace local (não publicado no npm), esse `npm install` tentou baixá-lo do registry público e falhou com `404 Not Found - GET https://registry.npmjs.org/@orion%2fshared`. Isso é fundamentalmente diferente da Vercel, que tem detecção nativa de monorepo no passo de install. Fix: Root Directory **vazio/raiz do repo**, com Build Command e Start Command explicitamente qualificados com `--workspace=apps/api` (ver comandos completos em §16.2) — não é o comando de start que quebrava, é especificamente o install atrelado ao Root Directory.
- **Vercel: um campo de env var mostrando um texto cinza "de exemplo" não significa que o valor real está visível/salvo** — pode ser um placeholder de uma variável genuinamente vazia (recém-criada) ou, mais enganoso ainda, o comportamento deliberado da Vercel de nunca reexibir o valor real de uma variável marcada como "Sensitive" depois de salva (mostra sempre um texto com estilo de placeholder no lugar). Isso causou um `500 MIDDLEWARE_INVOCATION_FAILED` em produção: `apps/web/src/lib/supabase/middleware.ts` chama `createServerClient(NEXT_PUBLIC_SUPABASE_URL, NEXT_PUBLIC_SUPABASE_ANON_KEY, ...)` dentro do middleware do Next.js (roda em toda requisição) — se qualquer uma dessas duas vier vazia/errada, `@supabase/ssr` lança de forma síncrona, e o Next.js transforma isso num 500 já na primeira tela. O valor não tinha sido salvo corretamente da primeira vez. Fix: apagar e recriar as duas variáveis com o toggle **"Sensitive" desligado** (não faz sentido mesmo para `NEXT_PUBLIC_*`, que já vão para o bundle JS público de qualquer forma) e colar o valor real de novo, depois redeploy.
- **Railway: confiar na porta-alvo que ela auto-detecta, não no fallback hardcoded do código** — ao configurar o domínio customizado `api.orionflightlab.com.br`, a Railway pediu uma porta-alvo e auto-detectou `8080` ("Railway magic", rotulado "8080 (node)" na UI), mesmo `apps/api/src/main.ts` tendo um fallback de `3001` quando `process.env.PORT` não está setado (`const port = process.env.PORT ? Number(process.env.PORT) : 3001`). Isso é esperado: a Railway injeta sua própria env var `PORT` no container em runtime, então a aplicação escuta na porta que a Railway atribuiu (8080 nesse caso), não no fallback do código — a porta auto-detectada pela Railway é a correta a usar, não o número que aparece hardcoded no `main.ts`.

## 15. Metodologia de teste ponta a ponta (replicar em sessões futuras)

Para validar qualquer endpoint/regra de negócio contra a infra real:

1. Criar organização de teste via `mcp__claude_ai_Supabase__execute_sql` (sempre incluir `updated_at` nos inserts manuais — não tem default no banco).
2. Criar usuário real: `supabase.auth.admin.createUser({ app_metadata: { organization_id, roles: [...] } })` + inserir `user_profiles` com o mesmo `id` **via `execute_sql`** (o client `service_role` do `supabase-js` não tem GRANT de `INSERT` nessa tabela — ver §14).
3. Pegar token real: `supabase.auth.signInWithPassword(...)` com o client `anon`.
4. Subir API/web (`npm run dev:api` / `dev:web`), testar via `curl` com `Authorization: Bearer <token>` ou via Playwright (`chromium.launch`) para o frontend.
5. **Sempre limpar ao final**: apagar as linhas criadas (ordem inversa das FKs) e `supabase.auth.admin.deleteUser(...)`. Confirmar `select count(*) from organizations` = 0 antes de encerrar.
6. Rodar `npm run lint && npm run typecheck` antes de qualquer commit.

### 16.1. Como testar crons/jobs agendados ponta a ponta (sem esperar o schedule)

Os handlers de `@Cron(...)` (`academic.cron.ts`, `notifications.cron.ts`, `financial.cron.ts`) não são endpoints HTTP — para exercitá-los de verdade sem esperar `EVERY_DAY_AT_*AM`:

1. `npm run build:shared && npm run build --workspace=apps/api` (gera `apps/api/dist/`).
2. Criar um script Node temporário em `apps/api/` (fora de `apps/api/`, os `require()` de pacotes do workspace não resolvem — precisa rodar de dentro do diretório do app) que:
   - Carrega `.env` manualmente (parse simples de `KEY=VALUE`, não há `dotenv` como dependência do projeto).
   - `const app = await NestFactory.createApplicationContext(AppModule)`.
   - Pega o serviço/cron via `app.get(...)` e chama o método do handler diretamente (ex.: `academicService.updateExpiredStatuses()`, `notificationsCron.checkCourseInactivity()`).
   - Para inspecionar jobs do BullMQ sem que o `NotificationsProcessor` já consuma: `app.get(getQueueToken(NOTIFICATIONS_QUEUE))`, `queue.pause()` antes de rodar os crons, depois `queue.getJobs(['waiting','delayed','paused'])`.
3. Apagar o script ao final (não commitar) — foi só instrumentação de teste, não faz parte do produto.
4. Fixtures de banco via `execute_sql` direto, com os valores exatamente nos limiares das RNs (ex.: `last_activity_at = now() - interval '190 days'` para RN-20 suspender, `'160 days'` para só alertar).

### 16.2. Como foi feito o deploy de produção (Vercel + Railway + Cloudflare)

Registrado aqui em forma copiável para uma sessão futura não precisar redescobrir nada — especialmente a armadilha do Root Directory da Railway (ver também §14).

**Cloudflare (DNS)** — plano free, nameservers do domínio apontando para o Cloudflare (migrados do Registro.br, registros MX/TXT de e-mail preservados sem alteração). Registros criados:
- `CNAME www` → o hostname aleatório que a Vercel atribuiu ao domínio (ex.: `c83f406feb6cc445.vercel-dns-017.com` — a Vercel não usa mais o antigo `cname.vercel-dns.com` genérico, gera um alvo por domínio/projeto; ver §14), proxy status **"DNS only"** (nuvem cinza, não laranja).
- `CNAME @` (apex/raiz) → mesmo padrão, via CNAME flattening do Cloudflare (permite CNAME na raiz da zona, o que DNS padrão não permite); a Vercel sugeriu redirect apex → `www` automaticamente, aceito.
- `CNAME`/`TXT` para `api.orionflightlab.com.br` conforme exigido pelo fluxo de domínio customizado da Railway (routing + verificação de propriedade — a Railway pede os dois registros, diferente da Vercel que só pediu o CNAME).

**Vercel (`apps/web`)** — projeto importado do GitHub (`sabrinadeccache/orion-flight-lab`), preset Next.js auto-detectado.
- Root Directory: `apps/web`
- Build Command (precisou de override — o `next build` default falha porque `packages/shared` ainda não foi compilado nesse ponto do monorepo):
  ```
  cd ../.. && npm run build:shared && cd apps/web && npm run build
  ```
- Install Command / Output Directory: **default da Vercel, sem override** — a Vercel tem detecção nativa de monorepo com npm workspaces e já roda `npm install --prefix=../..` sozinha (instala a partir da raiz real do repo, hoisting correto de `node_modules`). Funcionou de primeira, sem precisar mexer.
- Env vars (Production e Preview): `NEXT_PUBLIC_API_URL`, `NEXT_PUBLIC_SUPABASE_URL`, `NEXT_PUBLIC_SUPABASE_ANON_KEY`, `NEXT_PUBLIC_SENTRY_DSN`. Ver §14 sobre o toggle "Sensitive" nessas variáveis.
- Domínio: `www.orionflightlab.com.br` conectado via Vercel → Domains; a Vercel sugere redirect da raiz para `www`.

**Railway (`apps/api`)** — a Railway detectou o monorepo do GitHub sozinha e criou dois serviços por padrão, nomeados a partir do campo `name` do `package.json` de cada workspace (`@orion/api`, `@orion/web`); o `@orion/web` foi apagado (`apps/web` já está na Vercel), só `@orion/api` foi mantido.
- Root Directory: **vazio/raiz do repo** — não `apps/api` (ver a armadilha detalhada em §14; resumo: o passo de install da Railway roda antes do Build Command e é escopado ao Root Directory, então com `apps/api` como Root ele não enxerga o workspace e tenta baixar `@orion/shared` do registry público do npm, o que falha com 404).
- Build Command:
  ```
  npm install && npm run build:shared && npx prisma generate && npm run build --workspace=apps/api
  ```
- Start Command:
  ```
  npm run start --workspace=apps/api
  ```
- Watch Paths: `/apps/api/**` e `/packages/shared/**` (dois padrões — sem o segundo, uma mudança só em `packages/shared` não dispara redeploy da API).
- Env vars: `DATABASE_URL`, `DIRECT_URL`, `SUPABASE_URL`, `SUPABASE_SERVICE_ROLE_KEY`, `REDIS_URL`, `SENTRY_DSN`, `CORS_ALLOWED_ORIGINS=https://www.orionflightlab.com.br`, `NODE_ENV=production`.
- Domínio: subiu primeiro no domínio temporário gerado pela Railway (`*.up.railway.app`), depois `api.orionflightlab.com.br` via Networking → domínio customizado (porta-alvo e registros DNS — ver §14). Depois de validado, `NEXT_PUBLIC_API_URL` na Vercel foi atualizado do domínio temporário para `https://api.orionflightlab.com.br` e o `apps/web` foi redeployado.

Qualquer novo serviço adicionado à Railway a partir deste monorepo deve seguir o mesmo padrão (Root Directory vazio, Build/Start Command qualificados com `--workspace=<pacote>`).

## 17. Testes automatizados e CI

A validação manual ponta a ponta (§15) prova que uma RN funciona *no momento em que foi testada* — não protege contra uma mudança futura quebrá-la silenciosamente. Esta seção existe para isso: uma rede de segurança automática, que roda sozinha a cada push/PR via GitHub Actions (`.github/workflows/ci.yml`), sem depender de infraestrutura real compartilhada.

### Duas camadas de teste, dois bancos diferentes

- **Unit (`apps/api/src/**/*.spec.ts`, rodado com `npm run test --workspace=apps/api`)** — um arquivo por service/cron que implementa alguma RN, `PrismaService` **mockado** (`jest.fn()` por model/método usado). Cobre cada RN nos dois sentidos: o cenário que ela **bloqueia** (deve lançar `BadRequestException`/`NotFoundException` citando o número da RN) e o que ela **libera**. Hoje são 12 arquivos, 149 testes, cobrindo RN-05 até RN-32 inteiras (mapa completo: `academic`, `personnel`, `sgq`, `sgso`, `contracts`, `crm`, `financial` (+ seu cron), `notifications.cron`, `audit-log.interceptor`, `training` (conteúdo Segment→Material, IDOR de cada nível, regras de `MaterialType`, upload) e `lms` (IDOR de matrícula/lição/quiz/material, reaproveitamento de `registerAttendance`, cálculo de nota do quiz, resposta sem `is_correct`)) — `sgq`/`sgso` ganharam casos extras para os `find*` com campos derivados (`canClose`, `isHighRisk`/`canChangeStatus`) e para os `update*` novos (`updateAuditProgram`, `updateAudit`, `updateHazard`, `updateRisk` — este último cobre o recálculo de `risk_level`).
- **Integration (`apps/api/test/*.integration.spec.ts`, rodado com `npm run test:integration --workspace=apps/api`)** — usa o `PrismaClient` **real** (não mockado) contra um **Postgres efêmero**: no CI, um container `postgres:16-alpine` subido só para aquele job, com `prisma migrate deploy` aplicado; localmente, só roda se `DATABASE_URL` estiver setado (senão o describe inteiro é pulado via `describe.skip`) — **nunca aponte isso para o Supabase real do `.env`**, que deve continuar sempre vazio (§11). Hoje cobre a classe de bug do IDOR sistêmico (§13.9): criar uma entidade na organização B apontando para um recurso pai que pertence à organização A deve ser rejeitado, testado de verdade contra constraints reais do Postgres, não assumido.

Isso **não substitui** a validação manual ponta a ponta contra o Supabase real (§15) para mudanças grandes — são complementares. O CI garante que nada regride silenciosamente; a validação manual prova que a peça nova funciona de verdade contra a infra real na primeira vez.

- **E2E smoke test (`apps/web/e2e/*.spec.ts`, rodado com `npm run test:e2e --workspace=apps/web`, Playwright)** — a única camada de teste commitada que exercita a aplicação **viva** (browser real → `apps/web` real → `apps/api` real → Postgres real), diferente de unit (Prisma mockado) e integration (Prisma real, mas sem HTTP/Auth). Roda contra uma stack **local e efêmera do Supabase** subida via Supabase CLI (`supabase start` — Postgres + Auth/GoTrue + Storage + gateway, ver `supabase/config.toml`), nunca o projeto Supabase real: o guard de autenticação da API só aceita JWT real verificável via JWKS (sem bypass, de propósito — não vamos enfraquecer isso só para testar), e o projeto real está no plano Free, que provavelmente bloqueia branching efêmero (confirmado indiretamente ao checar `leaked password protection`, ver §11). `apps/web/e2e/global-setup.ts` provisiona uma `Organization` + um usuário `ADMIN` reais contra esse Supabase local antes dos testes (mesmo padrão do §15, sem teardown — o container inteiro é descartado ao fim do job). Cobertura deliberadamente mínima (é um *smoke* test, não uma suíte E2E completa): `/login` renderiza e protege `/dashboard` sem sessão (`auth.spec.ts`); login real pela UI → dashboard sem erros de console → criar um `Client` via formulário real → aparece na listagem (`golden-path.spec.ts`).

### Por que `supabase/rls/`, não `supabase/migrations/`

`supabase/rls/*.sql` (RLS, policies, triggers, buckets — antes vivia em `supabase/migrations/`) pressupõe que as tabelas de negócio já existem, e essas tabelas são criadas pelo **Prisma**, não pelo Supabase CLI. Como `supabase start`/`supabase db reset` aplica automaticamente qualquer coisa em `supabase/migrations/` contra um Postgres recém-criado — antes do Prisma ter chance de rodar —, manter esses arquivos nesse path quebraria a primeira subida local (RLS/policy em tabela inexistente). O diretório foi renomeado para `supabase/rls/` (ver `supabase/rls/README.md`) justamente para essa dependência de ordem ficar visível na estrutura do repo em vez de escondida — a sequência correta (`supabase start` → `prisma migrate deploy` → aplicar `supabase/rls/*.sql` manualmente) é a mesma tanto no job `test-e2e` do CI quanto para qualquer contribuidor futuro rodando a stack local pela primeira vez.

### Pipeline (`.github/workflows/ci.yml`)

Gatilho: `push`/`pull_request` para `main`. 6 jobs, todos independentes entre si (rodam em paralelo): `lint`, `typecheck`, `test-unit`, `test-integration` (com o serviço Postgres efêmero acima), `test-e2e` (Supabase local via CLI + Redis efêmero + Playwright, ver acima), `build` (compila `apps/api` e `apps/web`; usa variáveis `NEXT_PUBLIC_*` placeholder já que o build do Next faz prerender e o client do Supabase lança erro se a URL vier vazia — nenhuma delas precisa ser real, o build nunca faz uma chamada de rede de verdade).

### Rodando localmente

```bash
npm run test --workspace=apps/api              # unit — sempre roda, sem dependências externas
DATABASE_URL=postgresql://user:pass@localhost:5432/postgres \
DIRECT_URL=postgresql://user:pass@localhost:5432/postgres \
  npx prisma migrate deploy && \
  npm run test:integration --workspace=apps/api  # integration — precisa de um Postgres real (local/container), nunca o do .env
```

Docker não está disponível neste sandbox (§11), então nem o teste de integração nem o E2E puderam ser exercitados localmente nesta sessão — ambos foram validados por checagem de tipos isolada (`tsc --noEmit` nos arquivos novos) mais lint/typecheck do monorepo inteiro, e a prova real de que funcionam é o próprio job (`test-integration`/`test-e2e`) passando no GitHub Actions. Rodar o E2E localmente (numa máquina com Docker) segue o mesmo roteiro do job `test-e2e`: `supabase start` → `npx prisma migrate deploy` → aplicar `supabase/rls/*.sql` (ver `supabase/rls/README.md`) → subir `apps/api`/`apps/web` apontando pro Supabase local → `npm run test:e2e --workspace=apps/web`.

### Portal do aluno (LMS) — `/portal`, só role `ALUNO`, tema escuro "voo noturno"

Diferente do resto da tabela acima (telas de staff, tema claro), o portal é a área do próprio aluno — sem menu de staff, confinado pelo `middleware.ts` (ver "Navegação" acima), e com um tema visual próprio (escuro, inspirado em painel de instrumentos de cabine à noite — cores/fontes em `tailwind.config.ts` sob a chave `portal`, carregadas só em `apps/web/src/app/portal/layout.tsx`, não vazam pro resto do app). Sem editar/excluir em lugar nenhum — o aluno consome conteúdo e progride, não administra. O componente `ProgressRing` (`apps/web/src/components/portal/progress-ring.tsx`, mostrador circular estilo instrumento) é o elemento de assinatura, reaproveitado nas três telas e no preview do admin.

| Tela | Conteúdo | Observação |
|---|---|---|
| `/portal` | Hero "continuar de onde parou" (matrícula mais recente) + grid de cards de curso (capa em gradiente determinístico por código, `ProgressRing` no canto) | `GET /lms/my-enrollments` |
| `/portal/enrollments/[id]` | Capítulos expansíveis (Segment→Lesson) só leitura, `ProgressRing` em miniatura por lição, marca "quiz" quando a lição tem quiz | `GET /lms/enrollments/:id/content`; 404 se a matrícula não for do próprio aluno (IDOR) |
| `/portal/lessons/[id]` | Visualizador "modo cinema": `TEXTO` renderiza `content_html`, `VIDEO_EXTERNO` é um painel escuro com link "▶ Assistir vídeo", `ARQUIVO` tem botão "Abrir arquivo" (signed URL sob demanda, mesmo padrão de abrir a aba antes do `await` — ver §14); quiz inline em cartão âmbar se `hasQuiz`; botão "Marcar como concluída" em pílula | `GET /lms/lessons/:id/view` (tudo numa chamada), `POST /lms/lessons/:id/progress`. Concluir a lição reaproveita `AcademicService.registerAttendance` — não duplica a lógica da RN-05 |

**Preview do admin ("Visualizar como aluno")** — `/courses/[id]/preview` e `/courses/[id]/preview/lessons/[lessonId]` (staff only, `STAFF_ROLES`), linkado a partir de `/courses/[id]/content` e `/courses/[id]/progress`. Reaproveita o mesmo tema/componentes escuros do portal (`apps/web/src/app/courses/[id]/preview/layout.tsx` carrega as mesmas fontes), mas a sidebar continua clara (a sessão não é `ALUNO` puro). Dados vêm de `GET /training/courses/:id/preview-content`/`GET /training/lessons/:id/preview` — mesmo formato dos endpoints do aluno, mas sem `Enrollment`/progresso real (tudo `NAO_INICIADO`) e só leitura: sem marcar concluída, sem enviar quiz (só um aviso "esta lição tem um quiz formativo").

Convite pro portal: botão "Convidar para o portal" em `/students/[id]/edit` (`POST /students/:id/invite`), desabilitado até o aluno ter um `email` cadastrado; uma vez convidado (`user_profile_id` setado), o botão vira um aviso "já tem acesso" — sem reenvio de convite nesta rodada.

### Convenções das telas novas (seguir esse padrão ao adicionar mais)

- Toda tela de criação/edição é `'use client'`, com `fetch` direto pro `NEXT_PUBLIC_API_URL` (não usa `apiFetch` de `lib/api.ts`, que é só server-side) — mesmo padrão de `enrollments/new` e `exams/new`, que já existiam antes.
- Erro de validação: lê `body.errors[0].message` da resposta `{ data, meta, errors }` e mostra num `<p className="text-sm text-red-600">`.
- Exclusão sempre usa `confirm('Excluir ...? Essa ação não pode ser desfeita.')` antes de disparar o `DELETE` — não existe modal de confirmação customizado no projeto ainda, é o `confirm()` nativo do browser mesmo.
- Toda entidade "filha" que depende de uma "pai" que ainda não existe (ex.: Course sem Curriculum) mostra um aviso inline com link pra onde criar a dependência, em vez de deixar o formulário quebrar ou esconder o botão sem explicação.
- Toda tela `new`/`[id]/edit` tem um link `← Voltar para <Módulo>` (`text-sm text-slate-600 hover:underline`) logo acima do `<h1>`, apontando pra listagem do próprio módulo — não para `router.back()`, sempre um caminho fixo conhecido.
